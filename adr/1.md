Building log structure merge tree with write ahead log and leveled compaction.   

### 1 External contract
1. Open - takes session info and returns a db or error  
1. Get - takes key of []byte and returns a value of []byte. may add options later  
1. Has - takes a key -of []byte and returns a true if key is present  
1. Put - takes a key and value of []byte, returns error  
1. Delete - takes key, returns error  
1. NewScanner/Iterator - returns state machine  
    - Next - if next record, loads and returns true  
    - Key - returns next key  
    - Value - returns next value  
    - Release - let go of all resources  
1. Close - drain and release all resources  

### 2 Black Boxing
Client facing
1. DB - record point to library and implements contract. coordinates all subcomponents  
Components  
1. WAL - append only, durably stores at least all mutations in memtable at any given time  
1. Memtable - in memory records, holds versions based on sequence number  
1. Scanner - starts at a key, iterates for n records. scans memory and storage. Must be released after use  
1. Storage - manages sstables that have been written to disk  
1. Version - in memory representation of lsm tree derived from storage.   created at startup and updated after each compaction  
1. Compactor - merges new sstables after layer 0, only keeps latest and tombstones, updates version  

- PUT  
db.Put(key,value) -> WAL -> Memtable -> Eventually Storage 
- Delete
db.Delete(key) -> WAL -> Memtable -> Eventually Storage 
- Get
db.Get(key) -> Memtable -> Version if miss
- Scanner
db.NewScanner(start, limit) -> Memtable+Version -> Scanner -> Scanner.Next()... -> Scanner.Release()
- Flush (background)
Memtable -> Version -> Storage -> New Version
- Compact (background)
Version -> Compactor -> Storage -> New Version

### 3 Invariants per feature and component
1. Records in memtable must always be sorted key, then version
1. Sequence number must increase on every mutation
1. Get will always return the value for a key with the latest 
1. WAL must always contain at least what is in the memtable
1. Memtable must always flush records to disk sorted.
1. SSTables must never exceed n bytes
1. SSTables after compaction must never overlap in keys (level 0 can overlap)
1. Version must maintain levels 1+ sorted in memory
1. Version must be updated after every flush or compaction
1. Level 0 may contain duplicated keys. levels 1+ must only have one record per key
1. Compaction must keep only the latest version of each key (sequence number), if a tombstone is present and no later level has a version of key, it must be removed

### 4 Failure modes
1. Put crashes before WAL completes writing - must recover to last valid entry, truncate corrupted entry
1. Put crashes after WAL but before added to memtable - must to recover to last WAL entry
1. Crash during flush before written to memory - recovers from WAL -> memtable -> reflush

### Build Sequence
Follow invariants above. Components may be built as a slice to fulfill invariants and features rather than building each component to completion
